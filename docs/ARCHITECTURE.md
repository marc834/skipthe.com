# Architecture

## One-line summary

Cron-driven Python batch: pull → prefilter → AI classify → store → publish HTML site + RSS.

## Data flow

```
sources.yaml ──┐
               ▼
        ┌────────────┐    ┌────────────────────┐    ┌──────────────────┐
        │ collectors │ →  │ keyword prefilter  │ →  │ SQLite (items)   │
        │ rss + web  │    │ (neighborhood.kw)  │    │ INSERT OR IGNORE │
        └────────────┘    └────────────────────┘    └────────┬─────────┘
                                                             │ rows where
                                                             │ ai_summary IS NULL
                                                             ▼
                                                    ┌──────────────────┐
                                                    │ OpenAI classifier│
                                                    │ JSON: include,   │
                                                    │ category, etc.   │
                                                    └────────┬─────────┘
                                                             │ UPDATE items
                                                             ▼
                                              ┌──────────────────────────┐
                                              │ publishers (regenerate)  │
                                              │  - generate_feeds  → XML │
                                              │  - generate_site   → HTML│
                                              └────────┬─────────────────┘
                                                       ▼
                                  data/feeds/<nbhd>/*.xml + app/site/<nbhd>/*.html
                                                       │
                                                       ▼
                                                  nginx serves
```

## Components

### Collectors (`app/collectors/`)

- `rss_collector.collect_rss(source)` — `feedparser.parse(url)`, takes top 25 entries.
- `webpage_collector.collect_webpage(source)` — `requests.get` + BeautifulSoup, harvests every `<a>` with text ≥ 12 chars, dedupes by URL, caps at 30. Crude on purpose; the AI moderator does the real filtering.

Each collector returns a list of dicts with: `source_name, source_url, title, url, raw_summary, published_at, credibility`.

New source types (e.g. Reddit JSON, Nextdoor scrape, Mastodon ActivityPub) get their own module here and a new `type:` value in `sources.yaml`.

### Prefilter (`main.py:basic_keyword_prefilter`)

Cheap substring match against the neighborhood's `keywords` list. Anything that matches goes to the DB; everything else is dropped before we spend OpenAI tokens.

### Storage (`app/db/database.py`)

SQLite at `data/items.sqlite3`. Single table:

```sql
items(
    id INTEGER PK,
    source_name TEXT,
    source_url TEXT,
    title TEXT NOT NULL,
    url TEXT NOT NULL UNIQUE,        -- dedup key
    raw_summary TEXT,
    ai_summary TEXT,                  -- NULL until classified
    neighborhood TEXT,
    category TEXT,
    credibility TEXT,
    include INTEGER DEFAULT 0,        -- 1 = publish
    reason TEXT,
    published_at TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
```

`upsert_item` is `INSERT OR IGNORE` (the URL UNIQUE constraint does the dedup). `update_ai_result` is the post-classification UPDATE.

### Classifier (`app/ai/classifier.py`)

OpenAI Chat Completions with `response_format={"type": "json_object"}`. Default model is `gpt-4o-mini`. The system prompt anchors behavior; the user message carries the neighborhood config and the item. See [`AI_MODERATION.md`](AI_MODERATION.md) for the full contract.

The pipeline only classifies items where `ai_summary IS NULL`, capped at 50 per run. This is the rate-limit / cost throttle.

### Publishers

Two static-output writers, both regenerated from scratch each run:

- **`app/rss/generate_feeds.py`** — one RSS 2.0 XML per category into `data/feeds/<neighborhood>/`. Today: hardcoded to `nocatee` — needs to take neighborhood as an argument before a second neighborhood ships.
- **`app/site/` HTML generator** — *not yet built*. Pipeline will write `app/site/<neighborhood>/index.html`, `app/site/<neighborhood>/<category>.html`, and a top-level landing page. Spec in [`WEBSITE.md`](WEBSITE.md).

### Static site (`app/site/`)

Plain HTML. No JavaScript framework, no SPA, no app server. nginx serves the directory. The browsable feed pages will be generated by the pipeline; the static pages (`about.html`, `contact.html`, `legal.html`) are hand-written.

## Tech stack

- Python 3 (3.11+ assumed)
- `feedparser`, `requests`, `beautifulsoup4`, `lxml`, `PyYAML`, `python-dotenv`, `openai`
- (Likely add) `jinja2` for HTML templating once the browsable site lands
- SQLite (file-based, no server)
- nginx (serves static HTML and feed XML)
- Google Cloud Compute Engine VM
- cron (drives the batch)

No queue, no broker, no app server, no ORM. Deliberately small.

## Why this shape

- **Static HTML + RSS as the product**: caches well, costs nothing to serve, no auth surface, no DB on the request path.
- **SQLite over Postgres**: single-writer batch, low volume, file-level backups via `cp`.
- **Cron over a long-running daemon**: easier to operate, easy to reason about, no process supervision needed.
- **AI as moderator, not author**: the classifier decides include/exclude and writes a one-sentence neutral summary. Original headlines and links are preserved; readers always click through to the source.
- **Pipeline writes the website**: keeps the request path zero-logic. Adding "show me the latest items" becomes "regenerate one HTML file every 30 minutes," not "stand up a web app."
